Introduzione ai workspace: https://www.terraform.io/language/state/workspaces

  1. Deve essere usato un backend state
  1. Non tutti i backend state supportano i workspace (GCP sì)
  1. Una volta definito il backend state occorre fare init
  1. Da questo momento esiste solo il workspace di default
  1. Solitamente si crea un subfolder /environments in cui si raccolgono i file .tfvars per ogni ambiente (es. variables-prod.tfvars; variables-test.tfvars, ecc.)
  2. terraform workspace new <nome wks> (es. prod)
  3. terraform workspace list si vedono i workspace e quello corrente è indicato con *
  1. terraform workspace select per cambiare workspace
  1. Una volta selezionato i lworkspace si può fare apply/delete
  1. terraform apply -var-file=./environments/terraform-dev.tfvars
  1. Nel bucket di backend viene creato un tfstate per wrkspace

Tipico workflow per ambienti:

terraform workspace list

terraform workspace new dev

terraform workspace new prod

terraform apply -var-file=./environments/terraform-prod.tfvars

terraform workspace select dev

Using Terraform Cloud

Place your credentials in a Terraform Cloud environment variable:

  1. Create an environment variable called GOOGLE_CREDENTIALS in your Terraform Cloud workspace.
  2. Remove the newline characters from your JSON key file and then paste the credentials into the environment variable value field. You can use the tr command to strip newline characters. cat key.json | tr -s '\n' ' '
  3. Mark the variable as Sensitive and click Save variable.

ZERO DOWNTIME DEPLOYMENT

(aggiornamenti agli applicativi e questi non causano un fermo del servizio all’utente, l’update è completamente trasparente)

B/G (Blue/Green): deploy della nuova versione in parallelo a quella vecchia, e poi “switch”.

  • deploy della nuova versione (1 replica), verifica di stabilità e se non muore, allora deploy completo e “switch”

CANARY:

  • Switch “parziale”: 20% traffico sulla nuova versione, 80% della vecchia versione

  • Serverless : completamente demandato al cloud (non vedi “infrastruttura esterna”)
  • Kubernetes: si fa tramite l’oggetto Deployment+Service (terraform si può comandare: Kubernetes provider)

Queste due precedenti architetture nascono con l’idea di rispondere brillantemente a questa esigenza
